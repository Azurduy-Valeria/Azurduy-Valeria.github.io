Preventing Injection Attacks

Injection attacks are one of the first things they teach in web security — probably because they're still one of the most common ways things go wrong, even today.

What is an injection attack?

The idea is simple: an attacker puts malicious code into an input field (like a login form or search box), and the application runs it as if it were a valid command. The app was expecting data — it got instructions instead.

The most classic example is SQL injection. Imagine a login form that builds a database query like this:

SELECT * FROM users WHERE username = '[what you typed]'

If you type admin' OR '1'='1, the query becomes always true and you get in without a real password. The database just did what it was told.

The application trusted what the user sent. That's the mistake.

Types of injection attacks

SQL InjectionManipulates database queries. Can expose, change or delete data
XSSInjects JavaScript into a page other users see. Can steal sessions
Command InjectionRuns shell commands through the app if it calls the OS directly
LDAP / XMLTargets directory services or XML parsers with crafted input

How to prevent them

Never trust user input. That's the rule. Everything that comes from outside the application — form fields, URL parameters, headers — should be treated as potentially dangerous.

Parameterised queries are the main fix for SQL injection. Instead of building a query by joining strings, you tell the database the structure first and pass the data separately. The data physically cannot become part of the command.

Input validation means checking that what arrives looks like what you expected — right format, right length, only the characters you allow. Reject anything that doesn't fit.

Output encoding is how you stop XSS. Before showing user-supplied content on a page, encode the special characters so the browser treats them as text, not as HTML or JavaScript.

Least privilege on your database user helps limit the damage if something does get through. If your app only reads data, the database account shouldn't have permission to delete tables.

What surprised me: injection is in OWASP's Top 10 list of web risks every single year — not because it's new or hard to understand, but because developers still make the same mistakes. The fixes are straightforward. The hard part is being consistent about applying them.
