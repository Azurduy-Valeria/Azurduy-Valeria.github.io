topic 05 · patches

Why Patches Matter

I used to ignore update notifications. "I'll do it later." Learning about patch management made me realise how bad of an idea that is.

What is a patch?

When a vulnerability is found in software — by the vendor, a researcher, or a bug bounty hunter — the vendor (ideally) releases a fix. That fix is a patch. It closes the weakness.

Here's the problem: the moment the patch is released, the vulnerability it fixes becomes public. Security bulletins, changelogs, CVE databases — it's all out there. Attackers read those too. They immediately start looking for systems that haven't applied the fix yet.

The patch being released is also the announcement that the vulnerability exists. The race starts then.

The WannaCry example

In May 2017, the WannaCry ransomware attack spread to over 200,000 systems in 150 countries. It exploited a Windows vulnerability — but Microsoft had already released a patch for it two months earlier. The affected systems just hadn't updated.

That's not a story about a sophisticated attack. It's a story about delayed patching. And it caused billions of dollars in damage.

Patch management isn't just "click update"

For individuals, yes, just keep things updated. But in organisations it's a process:

InventoryYou can't patch what you don't know exists. First, know what's running
PrioritiseCritical patches for actively exploited bugs go first. Others follow a schedule
TestPatches can break things. Test in staging before pushing to production
AutomateManual patching at scale is slow and error-prone. Automate wherever possible

How it connects to everything else

Patching is one layer in a bigger strategy. Even with good patching, you need monitoring to detect attacks, least privilege to limit damage, and regular testing to find what's still exposed. None of these replace each other — they work together.

How patching is evolving

With containers and cloud infrastructure, the model is shifting. Instead of patching a running server, you rebuild it from a clean image with the fix applied. "Replace" instead of "repair." It's faster and more reliable when done right.

But the core idea stays the same: known vulnerabilities need to be addressed quickly. That's not going to change.

My honest reflection: I used to dismiss updates as annoying. Now I understand that every delayed update is a window where a known vulnerability sits open. It's not about being paranoid — it's just closing doors that don't need to stay open.
