Analytic Tools: Static and Dynamic Analysis

When I first heard "static analysis" and "dynamic analysis" I assumed they were very complex. They're actually pretty logical once you break it down.

What's the difference?

Static analysis looks at the code without running it. You're reading through the source code searching for things that look dangerous — passwords hardcoded, inputs that aren't checked, functions known to be risky. The program never actually executes.

Dynamic analysis runs the program and watches what it does. Instead of reading the recipe, you're watching someone cook and seeing if anything goes wrong in real time.

Static = read the code. Dynamic = run it and watch.

Why do we need both?

Static analysis is fast and can scan a lot of code quickly. But it doesn't know how the program will actually behave when it runs — it can flag things that turn out to be fine, and it might miss things that only appear at runtime.

Dynamic analysis sees real behaviour. But it can only find bugs in the code paths it actually triggers. If a bug hides in a feature that wasn't tested, dynamic analysis won't catch it.

So you use both — they cover each other's blind spots.

A bit of history

Static analysis goes back to the 1970s, when compilers started catching basic coding errors automatically. At first it was just syntax — "this line doesn't make sense." Over time the tools got smarter and started understanding security-specific patterns.

Dynamic analysis came out of the need to test real behaviour. Fuzzing — basically throwing random weird input at a program to see if it crashes — became popular in the 90s and is still used constantly today.

The main tool categories

SASTStatic Application Security Testing. Reads your source code. Tools: SonarQube, Semgrep
DASTDynamic Application Security Testing. Tests the running app. Tools: OWASP ZAP, Burp Suite
FuzzingSends unexpected input to find crashes. Tools: AFL, libFuzzer
IASTInteractive — sits inside the running app and watches from the inside. Best of both worlds

How this fits into real development

In modern teams these tools get plugged directly into the CI/CD pipeline — every time someone pushes code, automated scans run automatically. Security checks happen in the background without anyone having to remember to do them manually.

As a developer myself, this made a lot of sense to me. Security shouldn't be something you bolt on at the end — it should be baked into the workflow from the start.

My takeaway: Neither tool type is better. Static analysis is your first pass while writing code. Dynamic is your check once it's running. Use both.
